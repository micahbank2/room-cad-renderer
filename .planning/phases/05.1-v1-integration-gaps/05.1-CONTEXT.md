# Phase 5.1: v1.0 Integration Gaps - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning
**Type:** Gap closure (from v1.0-MILESTONE-AUDIT.md)

<domain>
## Phase Boundary

Close three concrete integration gaps identified in the v1.0 milestone audit:
1. **SAVE-02 startup hydration** — auto-save writes to IDB but no reload path
2. **Walk-camera room-switch respawn** — camera stranded when switching rooms in walk mode
3. **Orbit camera position restore** — position not restored on walk→orbit toggle (target is)

Scope is strictly these three fixes — no new features, no scope expansion.

</domain>

<decisions>
## Implementation Decisions

### Fix 1: SAVE-02 Startup Hydration
- **D-01:** Add a `useEffect` in `src/App.tsx` that runs once on mount (after productStore.load) and calls `listProjects()` → picks the most recent (by `updatedAt`) → `loadProject(id)` → `useCADStore.getState().loadSnapshot(project.snapshot)` + `useProjectStore.getState().setActive(id)`.
- **D-02:** If `listProjects()` returns an empty array (fresh install, no saved projects), do nothing — the existing cadStore default state ("Main Room") is correct starting state.
- **D-03:** `loadSnapshot` already calls `migrateSnapshot` defensively (Phase 5 D-15), so v1 legacy snapshots auto-upgrade on load.
- **D-04:** Do NOT mount `ProjectManager` component. The orphaned component stays deleted-in-spirit; a simple startup effect is cleaner than UI chrome the user doesn't need for v1.0. (ProjectManager stays on disk for later use if a "switch projects" UI is ever added.)
- **D-05:** Hydration happens AFTER `productStore.load()` so products are available when rooms render. Ordering: productStore.load() → wait for it → loadLastProject().

### Fix 2: Walk Camera Respawn on Room Switch
- **D-06:** Replace `didSpawn.current` gate in `src/three/WalkCameraController.tsx` with a `useEffect` keyed on `activeRoomId` (from `useUIStore` or `useCADStore`). Each time active room changes, center the camera at `{ x: room.width/2, y: 5.5, z: room.length/2 }` with yaw 0 (matches Phase 4 D-10 original spawn decision).
- **D-07:** Preserve first-mount behavior — useEffect with `[activeRoomId]` deps runs on mount AND on subsequent room changes, so no special-case for first-spawn needed.
- **D-08:** Only re-center position/yaw; do NOT re-lock pointer or reset any other walk state.

### Fix 3: Orbit Camera Position Restore
- **D-09:** In `src/three/ThreeViewport.tsx`, the `orbitPosRef` is already written by the OrbitControls `onChange` handler but never read back. Add a `useEffect` on the OrbitControls mount that, if `orbitPosRef.current` is non-null, calls `camera.position.set(...orbitPosRef.current)` and `orbitControls.update()`.
- **D-10:** Initialize `orbitPosRef.current` to the default camera spawn position `[halfW + 15, 12, halfL + 15]` on first mount so the ref is non-null from the start. Or alternatively: set the ref to the Canvas camera's initial position as soon as OrbitControls mounts.

### Claude's Discretion
- Where exactly to put the hydration `useEffect` in App.tsx — after productStore load effect, before the hasStarted detector.
- Whether to add a brief loading state while hydrating (recommendation: no, IDB reads are fast; existing default state renders fine until hydration completes).
- Exact signature changes to WalkCameraController internals (just replace gate — no API changes).
- Whether the orbit position restore should use `camera.position.copy(new THREE.Vector3(...))` or `camera.position.set(x, y, z)` — same effect, pick either.
- Whether to also expose a "reset camera" action (recommendation: no, not in scope).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`listProjects()`, `loadProject(id)`** — Already exported from serialization.ts, used by orphaned ProjectManager.
- **`loadSnapshot()`** — Already handles v1→v2 migration and pushes no history (transient load).
- **`useProjectStore.setActive(id)`** — Already exists.
- **`migrateSnapshot()`** — Already defensive; no new migration work.

### Established Patterns
- **Store-driven state:** All fixes update store state, components re-render reactively.
- **useEffect with activeRoomId dep:** Matches ThreeViewport's existing cameraMode effect pattern.
- **Ref-based camera persistence:** Already established in ThreeViewport for orbit target; extend to position.

### Integration Points
- **App.tsx top-level effect:** Add `loadLastProject()` async function called from useEffect after productStore.load completes (or in sequence).
- **WalkCameraController.tsx:** Swap `useRef(false) + if (!didSpawn.current)` for `useEffect(() => centerCamera(), [activeRoomId])`.
- **ThreeViewport.tsx:** Add `useEffect` or initialization logic to apply `orbitPosRef.current` to camera on OrbitControls mount.

</code_context>

<specifics>
## Specific Ideas

- Hydration chooses **most recent** project by `updatedAt` timestamp (sorted DESC in listProjects).
- No UI spinner during hydration — IDB read is fast.
- ProjectManager stays orphaned (not mounted, not deleted).
- Walk respawn re-centers at `{ room.width/2, 5.5, room.length/2 }` — same as Phase 4 D-10.
- Orbit position restore initializes ref to `[halfW + 15, 12, halfL + 15]` on first render.

</specifics>
