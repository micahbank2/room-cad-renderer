<objective>
Bind the new aoMap and displacementMap THREE.Texture refs (from Plan 01's useResolvedMaterial extension) to actual `meshStandardMaterial` instances across all surface meshes that consume the Material catalog, AND set up the uv2 attribute on every geometry that consumes aoMap. This is where Jessica finally SEES the upgrade — grout lines pop, fabric has depth, stone looks real.

Purpose: PBR-03 is the visible payoff of Phase 78. Without uv2, aoMap silently fails (RESEARCH.md Pitfall 1 — confirmed Three.js gotcha). Without `displacementScale={0.05}`, surfaces visibly explode (Pitfall 2 — 1ft default in foot-coordinate world). Both must land together for the rendering to work correctly.

**PBR-03 formal scope clarification (ADDED PER CHECKER REVISION):** PBR-03 as written in v1.20-REQUIREMENTS.md lists `WallMesh, FloorMesh, CeilingMesh, and ProductMesh` as the four in-scope sites. Audit of `src/three/ProductMesh.tsx` confirms ProductMesh does NOT consume the Material catalog — it branches on `product.gltfId` to render either `<GltfProduct>` or `<ProductBox>`, where ProductBox uses `useProductTexture(textureUrl)` for a single legacy product-image texture URL. There is no `finishMaterialId`, no `useResolvedMaterial` call, no `resolved.aoMap` available on the product render path. Therefore: **PBR-03's effective scope for Phase 78 is wall/floor/ceiling/custom-element surfaces ONLY**, where the Material catalog is already wired (Phase 67-69). ProductMesh and ProductBox receive an in-code scope-clarification comment so a future audit run does not flag them as missing PBR-03 sites; binding aoMap/displacementMap to product meshes is deferred until a future phase wires `finishMaterialId` onto products (analogous to walls). This decision is also recorded in this plan's truths and in 78-VERIFICATION.md scope notes when written.

Output: WallMesh, FloorMesh, CeilingMesh, CustomElementMesh bind aoMap + displacementMap props on their meshStandardMaterial and add uv2 setAttribute on the priority-1 geometries. Surfaces without these maps render identically to before. ProductMesh + ProductBox each carry a one-line scope-clarification comment.

Critical pattern — uv2 setup (RESEARCH §Pattern 3 + Pitfall 1):
```typescript
// Imperative useEffect approach (preferred for refs we already have):
useEffect(() => {
  const geo = meshRef.current?.geometry;
  if (!geo || geo.attributes.uv2) return;
  const uv = geo.attributes.uv as THREE.BufferAttribute | undefined;
  if (!uv) return;
  geo.setAttribute(
    'uv2',
    new THREE.BufferAttribute(uv.array.slice(), 2),
  );
}, []); // run once on mount
```

For declarative geometries (`<planeGeometry />`, `<boxGeometry />`), use the geometry ref-callback approach:
```tsx
<planeGeometry
  ref={(geo) => {
    if (geo && !geo.attributes.uv2) {
      const uv = geo.attributes.uv as THREE.BufferAttribute | undefined;
      if (uv) geo.setAttribute('uv2', new THREE.BufferAttribute(uv.array.slice(), 2));
    }
  }}
  args={[length, height]}
/>
```

WallMesh ExtrudeGeometry case: the priority-1 branch uses a SECONDARY `<planeGeometry>` overlay plane (line ~277), NOT the main ExtrudeGeometry. The aoMap is bound on the overlay plane, so uv2 setup happens on that planeGeometry. The base ExtrudeGeometry wall body renders neutral drywall — no Material binding, no aoMap, no uv2 needed.

Pattern #7 (StrictMode-safety, CLAUDE.md §StrictMode-safe useEffect cleanup): `geo.setAttribute('uv2', …)` writes to a Three.js geometry, not to a module-level registry. The geometry's lifetime is tied to the React component that renders it; on unmount Three.js disposes it. **No cleanup function needed** for the uv2 write — the buffer attribute is part of the geometry it lives on.
