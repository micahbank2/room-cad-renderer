# v1.15 Architectural Toolbar Expansion — Milestone Archive

## Goal

After v1.14 made the *furniture* real, v1.15 made the *room itself* richer. Jessica can now model her actual future home with all the architectural primitives a real floor plan needs.

## Phases

### Phase 59: Wall cutaway mode (CUTAWAY-01)
**Goal:** In 3D, when Jessica orbits to a side view, the wall closest to the camera should ghost so the room interior is visible from any angle.
**Requirements:** CUTAWAY-01
**Plans:** 1/1 complete (7 tasks)
**Built:** Three-mode state machine (`auto` / `off` / `manual`) in uiStore. `cutawayDetection.ts` (NEW): pure helper using **most-opposed-normal algorithm** + bbox-center sign convention + polar threshold (`Math.asin(-cameraForward.y) > 1.222 rad` = 70°-from-horizon auto-disable). Module-level scratch `THREE.Vector3` instances → zero per-frame allocation in the `useFrame` raycast loop. WallMesh `ghostMaterialProps(isGhosted)` helper spread on all 13 `<meshStandardMaterial>` JSX sites with constant `transparent: true` at construction (avoids Phase 49 BUG-02 shader-recompile pattern); only `opacity` + `depthWrite` animate. Per-room cutaway state in EXPLODE mode via `Map<roomId, wallId>`. Toolbar single-button cycle off ↔ auto. Right-click "Hide in 3D" / "Show in 3D" via Phase 53 menu. PR #142.

### Phase 60: Stairs (STAIRS-01)
**Goal:** New architectural primitive: stairs with configurable rise / run / width / step count, top-down 2D symbol with up-arrow, real stepped boxes in 3D.
**Requirements:** STAIRS-01
**Plans:** 1/1 complete (7 tasks)
**Built:** `Stair` as a NEW top-level entity (not a customElement.kind extension) — `RoomDoc.stairs: Record<string, Stair>` field. Defaults 7" rise × 11" run × 36" width × 12 steps (residential IBC R311). `stairTool.ts` (NEW): placement tool with **D-04 origin asymmetry** handling — `Stair.position` is bottom-step center, but `axisAlignedBBoxOfRotated` and `computeSnap` expect bbox-center; tool translates by `totalRunFt/2` along UP axis before snap, reverses on commit. `StairMesh.tsx` (NEW): N stacked `<boxGeometry>` per step + selection outline. **First Material Symbols D-33 allowlist expansion since Phase 33** — lucide-react has no Stairs icon; `Toolbar.tsx` (already on allowlist) + `TreeRow.tsx` (NEW exception with documented carve-out). **Snapshot version v3 → v4** with `stairs: {}` empty-maps migration. PR #143 (merged AFTER conflict-resolution with PR #144 due to a my branching error — see "Lessons learned" below).

### Phase 61: Openings — Archway / Passthrough / Niche (OPEN-01)
**Goal:** Extend the existing wall-opening codepath beyond doors and windows. Three new opening kinds: archway (rounded top), passthrough (full-height open doorway), niche (recessed wall display).
**Requirements:** OPEN-01
**Plans:** 1/1 complete (8 tasks)
**Built:** `Opening.type` enum extended to 5 kinds + optional `depthFt?: number` field for niche. **No snapshot version bump** — type-union extension + optional new field is back-compat. WallMesh kind-discriminated `THREE.Shape` builder: archway via `Path.absarc(centerX, shaftTopY, radius, 0, Math.PI, false)` for round top; passthrough = full-height through-hole; niche skips the hole entirely. `NicheMesh.tsx` (NEW): 5-plane recessed group on the **interior wall face** (uses Phase 59 outward-normal helper, INVERTED). **Critical sign-convention trap caught by research:** niche box center offsets from front face by `+N_out × d/2` (recess INTO wall along outward direction), NOT `+N_in × d/2`. Niche depth clamps to `wallThickness − 1"`. **D-11' (revised CONTEXT decision):** Phase 53/54 don't auto-inherit for openings — they were explicitly skipped at `FabricCanvas.tsx:498` (`// Skip: ... opening`). New code added: `ContextMenuKind` extended with `"opening"`, hit-test branch added, `getActionsForKind('opening')` 4-action menu, 2D overlay made `selectable + evented`. Toolbar **Wall Cutouts dropdown** (lucide `Frame` for niche + `RectangleHorizontal` for passthrough + Material Symbols `arch` for archway — `Toolbar.WallCutoutsDropdown.tsx` is the second new D-33 allowlist file). PR #144.

## Lessons learned (process)

1. **Verify PR merge state before branching the next phase.** Mid-milestone, I started Phase 61 from `origin/main` immediately after the user invoked `/gsd:discuss-phase 61` — but PR #143 (Phase 60) hadn't yet merged. Phase 61 went in ahead of Phase 60, leaving PR #143 with conflicts in 9 files. Recovery: merged main into Phase 60's branch, resolved conflicts (mostly straightforward — both phases extended the same union types). Going forward: explicit `gh pr view <N> --json state` check before branching any dependent phase.

2. **Stream timeouts on large phases are recoverable** if commits are atomic. Phase 60 + Phase 62 both hit Anthropic API stream timeouts mid-execution (~33-40 min, ~140 tool calls). Resume passes worked cleanly because each task was committed atomically — the resume just picked up at the next uncommitted task.

3. **Research catches assumption errors.** Phase 61 D-11 originally assumed Phase 53/54 "auto-inherit" for openings. Researcher checked the code and found openings were explicitly skipped in `FabricCanvas.tsx:498`. Saved a half-broken ship. Phase 62 applied the lesson preemptively — D-11 was scoped as NEW wiring from the start.
